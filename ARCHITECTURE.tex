\section{Component Reference}

Every file in TripMind is listed below, grouped by layer.
\textbf{API Path} = official API used when the corresponding \texttt{.env} key is set;
\textbf{Scraper Path} = default free path (no key required).

\section{Frontend Data Quality Controls}

When scrapers fail (blocked by anti-bot, JS not rendered, etc.), agents return ``Search Links'' fallback entries with generic names like ``Hotels in \{city\}'' and null prices. The frontend applies several filters to prevent these from reaching the user.

\subsection{Search Links Filtering}

\begin{longtable}{@{}L{3cm}L{4cm}L{6.5cm}@{}}
\toprule
\textbf{Location} & \textbf{Filter} & \textbf{Behavior when all filtered out} \\
\midrule
\endhead
Stays tab (verified hotels) & \texttt{source !== "Search Links" \&\& price\_per\_night} & Falls through to AI Suggested Options with name-specific Airbnb/Booking.com links \\
Budget tab (hotel options) & Same filter; require $\geq$2 remaining & Section hidden entirely; budget uses AI estimates \\
Budget tab (flight options) & \texttt{price \&\& source !== "Search Links"}; require $\geq$2 & Section hidden \\
Flights tab (verified) & \texttt{source !== "Search Links"} & Shows only AI Estimated Flights \\
Activities (verified match) & \texttt{source !== "Search Links"} & Shows AI Estimate badge instead of verified price \\
\bottomrule
\end{longtable}

\subsection{Price Trends Frontend Fallback}

The \texttt{PriceTrends.jsx} component guarantees charts are never empty:

\begin{enumerate}
\item Try backend data: \texttt{verification.price\_trends[destination].monthly\_prices}
\item If backend returned null/zero prices for all months, generate client-side estimates using seasonal shape curves (same \texttt{MONTH\_SHAPE} multipliers as the Python backend)
\item Fallback baselines: \$450/flight, \$120/hotel per night, scaled by travel month ratio
\item When using fallback, displays ``(seasonal estimates)'' label and adjusted disclaimer
\end{enumerate}

\subsection{Budget Selection System}

The Budget tab provides interactive hotel and flight option cards when verified data has $\geq$2 real results:

\begin{itemize}
\item \textbf{selectHotelOption(location, index)}: Switches selected hotel for a location, recalculates accommodation row in budget breakdown, updates total and per-person estimates
\item \textbf{selectFlightOption(index)}: Same for flights, multiplied by passenger count
\item \textbf{resetBudgetSelections()}: Restores original AI-estimated budget values
\item Delta badges show price difference (green = savings, red = more expensive) relative to currently selected option
\end{itemize}

\subsection{Booking Link Specificity}

URL builders in \texttt{constants.js} accept an optional \texttt{name} parameter to generate property-specific search links instead of generic city searches:

\begin{itemize}
\item \texttt{airbnbUrl(city, ci, co, adults, children, maxB, name)} --- searches ``\{name\}, \{city\}''
\item \texttt{bookingUrl(city, ci, co, adults, name)} --- same pattern
\item Activity links use \texttt{bookingKeyword + location} for Viator/GYG/TripAdvisor searches
\item Verified activities show direct \texttt{booking\_url} when the scraper/API found an exact match
\end{itemize}
