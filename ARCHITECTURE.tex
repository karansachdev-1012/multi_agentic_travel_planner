\section{Project Overview}

\subsection{What is TripMind?}

TripMind is an AI-powered travel planning application that generates personalized, multi-day trip itineraries using Anthropic's Claude large language model and then independently verifies every detail --- hotel prices, flight costs, activity availability, place existence, weather forecasts, safety advisories, currency rates, and more --- through 11 concurrent verification agents. The key architectural insight is the \textbf{dual-path design}: each verification agent can use either a paid official API (Amadeus, Booking.com, Viator, Google Places) when the user provides a key, or fall back to free web scraping and public APIs, meaning the app is fully functional at zero cost beyond Claude API credits.

\subsection{Three-Tier Architecture}

TripMind consists of three independently running services:

\begin{enumerate}
\item \textbf{Frontend} (React 19 + Vite, port 5173) --- A 9-step trip wizard that collects preferences (destinations, dates, budget, vibe, dietary needs, accessibility, children ages, passport country, etc.) and displays the generated itinerary across 7 tabs: Map, Itinerary, Stays, Flights, Budget, Suggestions, and Tips.

\item \textbf{Backend} (Node.js Express, port 3001) --- An API gateway that proxies requests to the Claude API for trip generation and to the Python service for verification. It handles rate limiting, Server-Sent Events (SSE) streaming, and serves the production build.

\item \textbf{Verification Service} (Python FastAPI + Playwright, port 8000) --- Runs 11 verification agents in parallel using \texttt{asyncio.gather()}. Each agent checks a SQLite cache first, then tries its data sources in priority order, and emits results back via SSE as they complete. Playwright provides headless Chromium for web scraping; httpx + BeautifulSoup handle lighter scraping; and five free public APIs require no keys at all.
\end{enumerate}

\subsection{End-to-End User Flow}

This section traces a complete user interaction from opening the app to seeing verified results.

\begin{infobox}[Phase 1: Trip Planning Form]
\begin{enumerate}[nosep]
\item User opens \texttt{http://localhost:5173} in the browser
\item The \texttt{FormSteps.jsx} component renders a 9-step wizard:
    \begin{itemize}[nosep]
    \item \textbf{Step 1} --- Origin city (where you are traveling from)
    \item \textbf{Step 2} --- Destination(s) (one or more cities/countries)
    \item \textbf{Step 3} --- Travel dates (check-in, check-out)
    \item \textbf{Step 4} --- Travelers (adults, children with ages)
    \item \textbf{Step 5} --- Budget (total budget, currency, max nightly hotel rate)
    \item \textbf{Step 6} --- Trip vibe (adventure, relaxation, cultural, nightlife, etc.)
    \item \textbf{Step 7} --- Extras (dietary restrictions, accessibility needs, pet-friendly, passport country)
    \item \textbf{Step 8} --- Activity level (relaxed, moderate, packed)
    \item \textbf{Step 9} --- Review all selections before generating
    \end{itemize}
\item All form state is managed by \texttt{TripContext.jsx} (React Context API)
\item User clicks ``Generate Trip'' on the review step
\end{enumerate}
\end{infobox}

\begin{infobox}[Phase 2: AI Trip Generation (Claude API)]
\begin{enumerate}[nosep]
\item \texttt{TripContext} calls \texttt{askClaudeStream()} from \texttt{src/api.js}
\item Frontend sends \texttt{POST /api/chat/stream} to Express (:3001)
\item Express proxies the request to the Anthropic Claude API with a structured prompt built by \texttt{src/prompts.js}
\item \textbf{Two-call strategy} to avoid token limits:
    \begin{itemize}[nosep]
    \item \textbf{Call 1 (Skeleton)}: Claude generates the trip structure --- destinations, daily overview, budget breakdown, hotel/restaurant/activity names, booking URLs, travel tips
    \item \textbf{Call 2 (Day-by-Day)}: Claude fills in the detailed day-by-day itinerary for each destination, using the skeleton as context
    \end{itemize}
\item Responses stream back via SSE; the \texttt{LoadingView.jsx} component shows animated progress with streaming text fragments
\item Claude's response is parsed as JSON and stored in \texttt{TripContext} state
\item The UI transitions from the form to the \texttt{TripGuide.jsx} multi-tab results view
\end{enumerate}
\end{infobox}

\begin{infobox}[Phase 3: Verification (11 Concurrent Agents)]
\begin{enumerate}[nosep]
\item Immediately after Claude's response is received, \texttt{TripContext} calls \texttt{requestVerification()}
\item Frontend sends \texttt{POST /api/verify/stream} with the complete trip plan + form data to Express
\item Express proxies to Python FastAPI (\texttt{POST /verify/stream} on port 8000)
\item The FastAPI endpoint calls \texttt{run\_verification\_streaming(req, on\_result)} in \texttt{orchestrator.py}
\item The orchestrator calls \texttt{\_build\_agent\_coros(req)} which constructs up to 20+ coroutines (some agents run once per destination):
    \begin{itemize}[nosep]
    \item Weather, Safety, Seasonal, Hotels: \textbf{one per destination}
    \item Flights, Price Trends: \textbf{primary destination only}
    \item Places, Activities, Reviews, Links: \textbf{batch} (all items from the plan)
    \item Currency: \textbf{single call} with all destinations
    \end{itemize}
\item All coroutines launch simultaneously via \texttt{asyncio.gather()}
\item Each agent independently: checks cache $\to$ tries API (if key set) $\to$ tries scraper $\to$ falls back to search URLs
\item As each agent completes, the orchestrator calls \texttt{on\_result()} which sends an SSE event back through Express to the browser
\item \texttt{VerificationBar.jsx} updates the progress bar in real time (e.g., ``7/11 agents complete'')
\item Results populate the tabs: weather appears in Tips, hotel prices appear in Stays and Budget, flight prices in Flights, etc.
\end{enumerate}
\end{infobox}

\begin{infobox}[Phase 4: User Interaction with Results]
\begin{enumerate}[nosep]
\item User browses the 7-tab results view:
    \begin{itemize}[nosep]
    \item \textbf{Map} --- Interactive map with pinned locations (verified via Places agent)
    \item \textbf{Itinerary} --- Day-by-day plan with activities, times, restaurant suggestions
    \item \textbf{Stays} --- Verified hotel options with prices, ratings, and booking links; ``Search Links'' fallback entries are filtered out
    \item \textbf{Flights} --- Verified flight options with airline, price, duration; AI estimates shown when scraper fails
    \item \textbf{Budget} --- Interactive budget breakdown; user can select different hotel/flight options and see live price delta badges (green = savings, red = more expensive) with automatic total recalculation
    \item \textbf{Suggestions} --- Alternative restaurants and activities
    \item \textbf{Tips} --- Weather forecast, safety advisory, seasonal insights, currency rates, visa requirements
    \end{itemize}
\item User can \textbf{swap hotels/restaurants} via the Swap Modal (triggers on-demand searches through \texttt{/api/hotels/search} or \texttt{/api/restaurants/alternatives})
\item User can \textbf{toggle currency} via CurrencyToggle (client-side conversion using verified rates)
\item User can \textbf{export to PDF} via the PdfExport button (lazy-loads \texttt{@react-pdf/renderer}, $\sim$1.5\,MB)
\end{enumerate}
\end{infobox}

\subsection{Agent Invocation Conditions}

Not all agents run for every trip. The orchestrator decides which agents to invoke based on the trip plan content and form data:

{\small
\begin{tabularx}{\textwidth}{@{}L{2.4cm}L{3cm}R@{}}
\toprule
\textbf{Agent} & \textbf{Condition} & \textbf{How Many Instances} \\
\midrule
Weather & Always (destinations exist) & One per destination city \\[4pt]
Safety & Always (destinations exist) & One per destination city \\[4pt]
Seasonal & Always (destinations exist) & One per destination city \\[4pt]
Currency & Always (destinations exist) & Single call (all destinations) \\[4pt]
Hotels & Plan contains accommodation locations & One per accommodation city \\[4pt]
Flights & Plan contains origin + destination & Single call (primary destination) \\[4pt]
Activities & Plan contains named activities & Single batch call (up to 15 activities) \\[4pt]
Reviews & Plan contains named places/restaurants & Single batch call (up to 10 places) \\[4pt]
Places & Plan contains named attractions & Single batch call (up to 20 places) \\[4pt]
Links & Plan contains URLs & Single batch call (up to 50 URLs) \\[4pt]
Price Trends & Plan contains origin + destination + dates & Single call (primary destination) \\
\bottomrule
\end{tabularx}}

\textbf{Key rules:}
\begin{itemize}[nosep]
\item If the Python verification service is not running, the app works fully with AI estimates only (graceful degradation)
\item If an agent times out (45\,s standard, 60\,s for price trends), it returns an error result but does not block other agents
\item If all data sources fail for an agent, it returns ``Search Links'' fallback entries which the frontend filters out
\item Agents with cached results return immediately without making any network calls
\item The \texttt{children\_ages} array is passed through to Hotels (group size) and Flights (passenger count); the \texttt{passport\_country} field is passed to Safety (advisory context)
\end{itemize}

# 1. Entry point (orchestrator.py:250)
async def run_verification_streaming(req, on_result):
    coros = _build_agent_coros(req)   # -> [("hotels", run_hotel_agent(...)), ...]

# 2. Agent dispatch (orchestrator.py:59)
async def run_agent_task("hotels", coro):
    result = await asyncio.wait_for(coro, timeout=45)

# 3. Hotel agent (agents/hotel_agent.py:11)
async def run_hotel_agent(city, check_in, check_out, ...):
    cached = await get_cached(cache_key)       # cache.py:get_cached
    if cached: return [HotelResult(**h) for h in cached]

    # Priority 1: Booking.com API
    config = get_booking_config()               # api_config.py
    if config["key"]:
        results = await search_hotels(...)      # api_clients/booking_api.py

    # Priority 2: Google Hotels scraper
    if not results:
        results = await scrape_google_hotels(...)  # scrapers/google_hotels.py

    # Priority 3: Booking.com scraper
    if not results:
        results = await scrape_booking(...)     # scrapers/booking_com.py

    # Priority 4: Search URL fallback
    if not results:
        results = [HotelResult(name="...", source="Search Links", ...)]

    await set_cached(cache_key, ...)            # cache.py:set_cached
    return results
\end{lstlisting}

\subsection{Price Trends Frontend Fallback}

The \texttt{PriceTrends.jsx} component guarantees charts are never empty:

\begin{enumerate}
\item Try backend data: \texttt{verification.price\_trends[destination].monthly\_prices}
\item If backend returned null/zero prices for all months, generate client-side estimates using seasonal shape curves (same \texttt{MONTH\_SHAPE} multipliers as the Python backend)
\item Fallback baselines: \$450/flight, \$120/hotel per night, scaled by travel month ratio
\item When using fallback, displays ``(seasonal estimates)'' label and adjusted disclaimer
\end{enumerate}

\subsection{Budget Selection System}

The Budget tab provides interactive hotel and flight option cards when verified data has $\geq$2 real results:

\begin{itemize}
\item \textbf{selectHotelOption(location, index)}: Switches selected hotel for a location, recalculates accommodation row in budget breakdown, updates total and per-person estimates
\item \textbf{selectFlightOption(index)}: Same for flights, multiplied by passenger count
\item \textbf{resetBudgetSelections()}: Restores original AI-estimated budget values
\item Delta badges show price difference (green = savings, red = more expensive) relative to currently selected option
\end{itemize}

\subsection{Booking Link Specificity}

URL builders in \texttt{constants.js} accept an optional \texttt{name} parameter to generate property-specific search links instead of generic city searches:

\begin{itemize}
\item \texttt{airbnbUrl(city, ci, co, adults, children, maxB, name)} --- searches ``\{name\}, \{city\}''
\item \texttt{bookingUrl(city, ci, co, adults, name)} --- same pattern
\item Activity links use \texttt{bookingKeyword + location} for Viator/GYG/TripAdvisor searches
\item Verified activities show direct \texttt{booking\_url} when the scraper/API found an exact match
\end{itemize}

\subsection{Price Trends Agent}
\begin{infobox}[Price Trends --- Dual-Path (Composite)]
{\small
\begin{tabularx}{\textwidth}{@{}L{3cm}R@{}}
\textbf{Function}     & \texttt{run\_price\_trends\_agent(origin, destination, check\_in, check\_out, adults=2, children=0, currency="USD", country\_code="")} \\[3pt]
\textbf{Strategy}      & Baseline scrape + seasonal multiplier estimation (2 scrapes, not 24) \\[3pt]
\textbf{Cache Key}     & \texttt{ptrends:\{origin\}:\{dest\}:\{adults\}:\{children\}} \\[3pt]
\textbf{Cache TTL}     & 24 hours \\[3pt]
\textbf{Output Model}  & \texttt{PriceTrendsResult} --- monthly\_prices (12 $\times$ MonthlyPrice: month\_name, avg\_flight, avg\_hotel, tourism\_level) \\[3pt]
\textbf{Inputs Used}   & \texttt{origin}, \texttt{destination}, \texttt{check\_in/out}, \texttt{adults}, \texttt{children\_ages} (length), \texttt{currency}, \texttt{country\_code} \\
\end{tabularx}}

\textbf{Three-step algorithm:}
\begin{enumerate}[nosep]
\item \textbf{Baseline fetch} --- scrape flight and hotel prices for the travel month only (Amadeus API $\to$ Google Flights; Booking API $\to$ Google Hotels; 35\,s timeout each)
\item \textbf{Seasonal estimation} --- multiply baseline by per-month shape curves:
    \begin{itemize}[nosep]
    \item Tourism multipliers: peak = 1.35, shoulder = 1.05, off-peak = 0.75
    \item Hemisphere-aware: southern countries (AU, NZ, AR, CL, BR, ZA, etc.) use inverted seasonal curves
    \item Fallback baselines if scraping fails: \$450/flight, \$120/hotel per night
    \end{itemize}
\item \textbf{Cache validation} --- skips stale entries where all prices are null
\end{enumerate}
This approach avoids 24 separate scrapes (12 months $\times$ flights + hotels) while still producing reasonable month-by-month estimates.
\end{infobox}

\section{Orchestrator Internals}

The orchestrator (\texttt{agents/orchestrator.py}) is the central dispatcher that runs all agents concurrently and streams results back to the frontend via Server-Sent Events (SSE).

\subsection{Dispatch Pipeline}

\begin{enumerate}
\item \textbf{\texttt{run\_verification\_streaming(req, on\_result)}} --- Main entry point. Builds agent coroutines, dispatches them in parallel, applies results, and calls \texttt{on\_result} after each agent completes.

\item \textbf{\texttt{\_build\_agent\_coros(req)}} --- Constructs a list of \texttt{(name, coroutine)} pairs:
\begin{itemize}[nosep]
\item \textbf{Per-destination agents}: Weather, Safety, Seasonal, Hotels --- one coroutine per destination (e.g., \texttt{"weather:Paris"}, \texttt{"hotels:Tokyo"})
\item \textbf{Single-destination agents}: Flights, Price Trends --- primary destination only
\item \textbf{Batch agents}: Places (up to 20), Activities (up to 15), Reviews (up to 10), Links (up to 50 URLs)
\item \textbf{Global agents}: Currency (single call with all destinations)
\end{itemize}
Destinations extracted from \texttt{req.destinations} or \texttt{trip\_plan.allDestinations}. Primary destination from \texttt{trip\_plan.primaryDestination}.

\item \textbf{\texttt{run\_agent\_task(name, coro)}} --- Wraps each coroutine with timeout enforcement:
\begin{itemize}[nosep]
\item Price trends: 60\,s timeout (needs baseline scrape + estimation)
\item All other agents: 45\,s timeout
\item On timeout: returns error result, does not crash other agents
\end{itemize}

\item \textbf{\texttt{asyncio.gather(*tasks, return\_exceptions=True)}} --- All agent tasks run in parallel. Exceptions are captured, not raised, so one failing agent does not block others.

\item \textbf{\texttt{\_apply\_result(response, name, result)}} --- Maps agent name to the correct field on \texttt{VerifyResponse}:
\end{enumerate}

\texttt{safety:\{dest\}} & \texttt{response.safety[dest]} & SafetyResult \\[3pt]
\texttt{seasonal:\{dest\}} & \texttt{response.seasonal[dest]} & SeasonalResult \\[3pt]

\subsection{Server-Sent Events (SSE) Streaming}

Server-Sent Events (SSE) is a standard HTTP mechanism that allows a server to push real-time updates to a client over a single, long-lived connection. Unlike WebSockets, SSE is unidirectional (server-to-client only), uses plain HTTP, and automatically reconnects on failure. TripMind uses SSE in two places: Claude trip generation (\texttt{/api/chat/stream}) and agent verification (\texttt{/api/verify/stream}).

The \texttt{on\_result(name, status, response)} callback is invoked after each agent completes. This enables real-time progress updates in the frontend:

\begin{enumerate}[nosep]
\item The FastAPI endpoint \texttt{/verify/stream} creates a \texttt{StreamingResponse} with \texttt{content-type: text/event-stream}
\item As each agent finishes, \texttt{on\_result} emits an SSE event containing the agent name, completion status, and the partial \texttt{VerifyResponse} accumulated so far
\item The React frontend receives these events via the browser's built-in \texttt{EventSource} API; the \texttt{VerificationBar.jsx} component updates its progress bar in real time as each agent reports in
\item When all agents have completed, a final SSE event delivers the complete \texttt{VerifyResponse} with all results, and the connection closes
\end{enumerate}

\subsection{Verification Score}

After all agents complete, the orchestrator computes a verification score:

$$\text{verification\_score} = \frac{\text{successful agents}}{\text{total agents}} \times 100$$

An agent is ``successful'' if it returns a result without an error field. Agents that time out or throw exceptions are counted as failures but do not prevent the response from being sent.
